## プロジェクト概要

Even Realities G2(HUD付きスマートグラス)上で動作するインターバルタイマーアプリ「TabaTimer」を開発する。タバタ式トレーニングを含む、運動/休憩のインターバルを繰り返すトレーニングを行うユーザーが、スマホや時計を手に取らずに視界内のHUDで残り時間やラウンド数を確認できるようにする。

### 目標

- **当面のゴール: evenhub-simulator 上で動作するインターバルタイマーを完成させる**(実機なしで動作確認できる状態)
- タイマー情報(残り時間、現在のフェーズ、ラウンド数)を G2 のディスプレイ領域に表示できる
- 運動時間・休憩時間・ラウンド数をカスタマイズしたインターバルタイマーを実行できる(タバタ式 20秒+10秒×8ラウンド を含む)
- 実機(Even G2)での動作確認は実機到着後(2026年8月頃見込み)の将来フェーズとする

### 背景

- Even G2 はカメラ非搭載・HUD表示特化のスマートグラス。トレーニング中(特に運動フェーズ中)はスマホや時計を確認する動作が困難であり、視界内に常時タイマーが表示されていればハンズフリーで確認できる
- Even Realities は公式アプリプラットフォーム **Even Hub** を提供しており(2026年4月ローンチ)、アプリは Web 技術(HTML/CSS/JS/TS)で開発し、スマホの Even App 内で実行され、グラスは表示レンダリングを担当する
- 公式シミュレータ `evenhub-simulator`(npm)があり、実機なしでローカル開発・動作確認ができる。G2 実機は注文済みで2026年8月頃到着見込みのため、それまでシミュレータを開発・検証環境とする
- このPC(Ubuntu 24.04)には Bluetooth アダプタがないが、Even Hub 構成ではグラスとの通信はスマホの Even App が担うため問題にならない。ユーザーのスマホは iPhone だが、アプリは Web 技術のため macOS/Xcode も不要

## ドメイン知識

- **Even G2**: Even Realities 社のスマートグラス。レンズ内の導光板に緑色モノクロのテキスト/グラフィックを表示する(両眼)。カメラ非搭載。スマホの公式アプリ(Even App)と BLE で接続して動作する
- **Even Hub**: Even Realities の公式アプリプラットフォーム兼ストア(https://hub.evenrealities.com/ 、ドキュメントは /docs 配下)。アプリの種類はプラグイン(バックグラウンド層)/ダッシュボードウィジェット/AIスキル統合の3種。アプリロジックはスマホの Even App 内で実行され、グラスは表示レンダリングとスクロール処理を担当する
- **Even Hub アプリの開発フロー**: Web 技術(Vite + SDK)でコード作成 → `evenhub-simulator` でローカル確認 → QR またはポータルで実機にサイドロード → `evenhub pack` で .ehpk 形式にパッケージ化 → Even Hub に提出(公開する場合)。プライベートビルド(個人専用アプリ)の配布も可能
- **グラス画面の基本仕様**: キャンバスは 576×288(左上原点、X右+/Y下+)。UI は「コンテナ」(list / text / image)の集合として構築する。SDK の制約値の詳細は `docs/issues/feat-001-hello-world/research.md` を参照
- **シミュレータの automation API**: `evenhub-simulator <URL> --automation-port <PORT>` で起動すると HTTP API が使える(`/api/screenshot/glasses` = グラス画面の 576×288 PNG、`/api/console` = WebView のコンソールログ、`/api/input` = タッチ操作送信、`/api/ping`)。Claude Code はこれを使い、ユーザーの手動テスト前に表示・動作を自律確認できる。targetUrl 引数は必須(省略するとエラー終了)
- **インターバルタイマー**: 運動(ワーク)と休憩(レスト)のフェーズを交互に繰り返すトレーニング用タイマー。各フェーズの秒数とラウンド数を設定できる
- **タバタ式トレーニング**: インターバルトレーニングの代表的プロトコル。「20秒の高強度運動 + 10秒の休憩」を8ラウンド(計4分)行う

### 技術知見ノート(docs/tech_notes.md)

## 開発方針

- **シンプルな機能を一つずつ作り、積み重ねて目的を達成する**
- 大きな機能を一度に作らない。小さく作って動作確認し、次の機能へ進む
- **ゴールやロードマップは固定しない**。初めての Even Hub 開発のため、案件の完了や新しい知見(SDK仕様の判明、実機到着)に応じて、本ファイルの目標・背景・技術スタックを都度見直して更新する

### 機能追加フロー(feat-XXX 案件)

新機能を追加する場合、以下のフローを**厳守**する。**planモードは使わない**(通常モードで調査・計画を行う)。

1. **案件作成** → `docs/issues/feat-{number}-{slug}/` フォルダを作成し、`docs/BACKLOG.md` に追加する
2. **調査・計画** → 通常モードで既存コードを調査し、要求仕様書(`docs/REQUIREMENTS_STANDARD.md` 準拠)と機能設計書(`docs/DESIGN_STANDARD.md` 準拠)を作成する
3. **ドキュメント保存** → 要求仕様書を `docs/issues/{案件フォルダ}/requirements.md`、機能設計書を `docs/issues/{案件フォルダ}/design.md` にファイル保存する。**保存が完了するまで実装に進んではならない**
4. **レビュー(Codex + 人)** → 保存されたドキュメントを **Codex** でレビューする。実行方法は後述の「Codexによるレビューの実行方法」を参照。ユーザーも同時にレビューする。レビュー実行時は `docs/REVIEW_CRITERIA.md` の基準に従うこと
5. **修正(必要な場合)** → レビューで問題があれば、再調査してドキュメントを更新する。**ステップ2〜4を問題がなくなるまで繰り返す**
6. **実装** → ドキュメント(要求仕様書・機能設計書・CLAUDE.md)を読んで実装する。実装完了後、「テスト」のルールに従ってテストを実行する
7. **手動テスト** → ユーザーがテストする。以下の問題があれば `docs/BUGFIX_STANDARD.md` に従って修正計画を `docs/issues/{案件フォルダ}/investigation.md` に追記する(上書きしない。イテレーション番号を付けて履歴を残す)。**ユーザーの承認を得た上で、ステップ2〜7を繰り返す**(コード修正はステップ6で行う。ステップ7で直接コードを編集してはならない)
   - 不具合の発見
   - 要求通りに実装されていない
   - 要求仕様作成時のヒアリング漏れ
8. **完了** → `docs/BACKLOG.md` のステータスを Closed に更新する。ファイルの追加・削除があった場合は `CLAUDE.md` のディレクトリ構成を最新に更新する。**この案件で得た再利用価値の高い技術的知見を `docs/tech_notes.md` に抽出・追記する**(出典を明記。再利用価値のある知見がなければ追記不要)

### 不具合修正フロー(bug-XXX 案件)

既存機能の不具合を修正する場合、以下のフローを**厳守**する。

1. **案件作成** → `docs/issues/bug-{number}-{slug}/` フォルダを作成し、`docs/BACKLOG.md` に追加する。`README.md` に不具合の概要と再現手順を記録する
2. **調査・修正計画** → `docs/BUGFIX_STANDARD.md` に従い、既存コードを調査する。修正計画を `docs/issues/{案件フォルダ}/investigation.md` に記録する。**この時点でコードを編集してはならない**
3. **ドキュメント保存** → investigation.md の保存を確認する。**保存が完了するまで実装に進んではならない**
4. **レビュー(Codex + 人)** → 保存されたドキュメントを **Codex** でレビューする。実行方法は後述の「Codexによるレビューの実行方法」を参照。ユーザーも同時にレビューする。レビュー実行時は `docs/REVIEW_CRITERIA.md` の基準に従うこと
5. **修正(必要な場合)** → レビューで問題があれば、再調査してドキュメントを更新する。**ステップ2〜4を問題がなくなるまで繰り返す**
6. **実装** → 承認された修正計画に沿ってコードを修正する。計画にない変更が必要になった場合は中断して報告する
7. **手動テスト** → ユーザーがテストする。問題があれば `docs/BUGFIX_STANDARD.md` に従って investigation.md にイテレーション番号を付けて追記し、**ユーザーの承認を得た上で、ステップ2〜7を繰り返す**(コード修正はステップ6で行う。ステップ7で直接コードを編集してはならない)
8. **完了** → `docs/BACKLOG.md` のステータスを Closed に更新する。ファイルの追加・削除があった場合は `CLAUDE.md` のディレクトリ構成を最新に更新する。**この案件で得た再利用価値の高い技術的知見を `docs/tech_notes.md` に抽出・追記する**(出典を明記。再利用価値のある知見がなければ追記不要)

### Codexによるレビューの実行方法

機能追加・不具合修正フローのステップ4(レビュー)では、Claude Code 自身が `codex exec` コマンドを実行して Codex にレビューさせる。Subagent は使わない。

使用するモデルは `~/.codex/config.toml` のデフォルト設定に従う。本ファイルのコマンドにはモデル指定(`-m`)を書かない。モデルを切り替えたい場合は `~/.codex/config.toml` を編集する(全プロジェクト共通で反映される)。

#### 初回レビュー(機能追加の場合)

```bash
codex exec "docs/REVIEW_CRITERIA.md の基準に従い、以下のドキュメントをレビューせよ: docs/issues/{案件フォルダ}/requirements.md docs/issues/{案件フォルダ}/design.md 。瑣末な点へのクソリプはしないで、致命的な点のみ指摘して。発見した問題を重要度(高/中/低)で分類し、修正提案とともに報告すること。"
```

#### 初回レビュー(不具合修正の場合)

```bash
codex exec "docs/REVIEW_CRITERIA.md および docs/BUGFIX_STANDARD.md の基準に従い、以下のドキュメントをレビューせよ: docs/issues/{案件フォルダ}/investigation.md 。瑣末な点へのクソリプはしないで、致命的な点のみ指摘して。発見した問題を重要度(高/中/低)で分類し、修正提案とともに報告すること。"
```

#### 再レビュー(共通)

ドキュメントを更新して再レビューする場合、最初のレビューの文脈を保持するため `resume --last` を使う:

```bash
codex exec resume --last "ドキュメントを更新したので再レビューして。前回と同じ基準で。瑣末な点へのクソリプはしないで、致命的な点のみ指摘して。重要度(高/中/低)で分類し、修正提案とともに報告すること。"
```

**注意**: `resume --last` を付けないと最初のレビューの文脈が失われる。

#### レビュー終了条件

重要度「高」「中」の指摘がなくなるまで、修正 → 再レビューを繰り返す。「低」のみになったら人レビューに進む。

#### この環境での注意(2026-06-12 判明)

- このPCでは Codex のサンドボックス(bubblewrap)が初期化できず(`bwrap: loopback: Failed RTM_NEWADDR: Operation not permitted`)、Codex は**リポジトリ内のファイルを自力で読めない**
- そのため、レビュー対象ファイル(REVIEW_CRITERIA.md 含む)の**全文をプロンプトに含めて渡す**こと
- 初回レビュー: `{ cat 対象ファイル...; } | codex exec "指示文"`(stdin の内容がプロンプトに添付される)
- 再レビュー: `codex exec resume --last` は **stdin を読まない**ため、`codex exec resume --last "指示文 ... $(cat 対象ファイル)"` のようにプロンプト引数へ直接埋め込むこと
- プロンプトには「コマンド実行は試みず、渡した内容だけに基づいてレビューすること」を明記する

- 重要度:高と中は修正対象とする

## Claude Code 運用ルール

### コミット・プッシュのルール

- コミット/プッシュはグローバルの `/git-commit` スキル(`~/.claude/skills/git-commit/SKILL.md`、全プロジェクト共通)で行う。コスト最適化のため、commit/push の実作業を **Sonnet の subagent** が実行する(スキル本体は subagent を起動するだけ。インライン `model: sonnet` 方式は長い会話で 1M コンテキスト課金ゲートに当たるため不可)
- `/git-commit` = コミットのみ、`/git-commit push` = コミット + push
- ユーザーが自然言語で「コミットして」と指示した場合も、このスキルを呼び出すこと
- **このリポジトリ固有の規約**(グローバルスキルはこの記述を読んで従う):
  - main ブランチに直接コミットする(ブランチは切らない)
  - コミットメッセージは日本語で書く
  - コミットメッセージ末尾に `Co-Authored-By: Claude Fable 5 <noreply@anthropic.com>` を付ける
  - Bash の `cd <path> && <command>` 連結は禁止(「Bash 実行時のルール」参照)

### 一時ファイル・調査作業のルール

- 調査・実験用の一時ファイルやお試しインストールは、システムの `/tmp` ではなく **プロジェクト内の `tmp/` ディレクトリ**(`/home/ogu4/git/TabaTimer/tmp/`)で行う
- `tmp/` は `.gitignore` で git 管理外にしてある
- `tmp/` の中身は使い捨て。削除してよいのは `tmp/` 配下のみで、それ以外の既存ファイルの削除・移動・上書きはユーザーの明示的な指示なしに行わない

### Bash 実行時のルール

- **`cd <path> && <command>` の連結は禁止。** Bashツールはプロジェクト作業ディレクトリで動くため `cd` は不要。連結すると先頭トークンが `cd` になり、`.claude/settings.json` / `.claude/settings.local.json` のallowlist(例: `Bash(codex exec *)`、`Bash(git status)`)が一致せず、毎回パーミッションプロンプトが発生する
- 別ディレクトリで実行する必要がある場合は、コマンド側のオプションを使う(例: `git -C <path> status`、`make -C <path> ...`)
- どうしても複数コマンド連結が必要な場合も、先頭トークンが安全・許可済みであるかを確認してから書く
