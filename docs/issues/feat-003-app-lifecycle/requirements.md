# feat-003 要求仕様書: アプリの起動・終了処理

`docs/REQUIREMENTS_STANDARD.md` に準拠。

## 1. プロジェクト概要

- **何を作るのか**: TabaTimer(Even Hub アプリ)の起動処理と終了処理を、公式 SDK 推奨シーケンスに沿って確定する。具体的には (a) ブリッジ取得 → 起動元判別(`onLaunchSource`)→ 初期画面生成、という起動フロー、(b) ダブルタップおよびライフサイクルイベントによる終了/後始末フロー、を仕様として定める。
- **なぜ作るのか**: feat-001/002 で起動・終了の挙動はコード上に存在するが仕様化されておらず、公式推奨の `onLaunchSource` も未実装。今後インターバルタイマー本体を載せる前に、アプリのライフサイクル骨格を確定し、再現性のある土台にする。
- **誰が使うのか**: TabaTimer を起動・終了するエンドユーザー(トレーニング者)。および本リポジトリの実装者(/clear 後の自分自身を含む)。
- **どこで使うのか**: 当面は `@evenrealities/evenhub-simulator`(グラス画面 576×288 を再現)。将来は Even G2 実機(2026年8月頃見込み)。

## 2. 用語定義

| 用語 | 定義 |
|------|------|
| ブリッジ | `EvenAppBridge`。WebView アプリとホスト(Even App / シミュレータ)間の双方向通信オブジェクト。`waitForEvenAppBridge()` で取得 |
| 起動元(launch source) | アプリがどこから開かれたか。`'appMenu'`(スマホ Even App メニュー)または `'glassesMenu'`(グラスメニュー)。`bridge.onLaunchSource` で1回だけ通知される |
| 初期画面生成 | `createStartUpPageContainer()` によるグラス画面コンテナの作成。アプリ起動後1回だけ有効 |
| 後始末(cleanup) | 更新ループの停止・イベント購読の解除など、終了時に行う資源解放処理 |
| ライフサイクルイベント | `onEvenHubEvent` の `sysEvent` で届く、前面化/背面化/終了などのOSイベント |
| 前面/背面 | アプリが他アプリの手前にあり描画対象である状態(前面)/裏に回った状態(背面) |

機能設計書・コードでも本表の用語を用いる。

## 3. 機能要求一覧

### FR-001 ブリッジ取得とタイムアウト

- **概要**: 起動時に `waitForEvenAppBridge()` でブリッジを取得する。`BRIDGE_TIMEOUT_MS`(8000ms)以内に取得できなければエラーログを出して起動処理を中止する。
- **入力**: ページロード(WebView による `main.ts` 実行)。
- **出力**: 成功時はブリッジオブジェクト取得・以降の起動処理へ進む。失敗時は `console.error` に「ブリッジ未取得・シミュレータ/実機で開け」という旨のメッセージを出力し、画面生成を行わない。
- **受け入れ基準**:
  - シミュレータで開くと8秒以内にブリッジを取得し、後続処理(FR-003)へ進む。
  - 普通のブラウザで開くと8秒後にエラーログが出て、それ以上進まない(クラッシュしない)。

### FR-002 起動元の判別とログ出力(新規)

- **概要**: ブリッジ取得直後(初期画面生成より前)に `bridge.onLaunchSource(callback)` を登録し、通知された起動元(`appMenu` / `glassesMenu`)をログ出力する。起動元による画面分岐は行わない。
- **入力**: ホストからの launch source 通知(ロード完了時に最大1回)。
- **出力**: `console.log('[lifecycle] launch source: <source>')`。通知が来ない環境では何も出力しない(エラーにしない)。
- **受け入れ基準**:
  - シミュレータが launch source を push する場合、コンソールに起動元(`appMenu` または `glassesMenu`)が1回出力される。
  - launch source の通知有無にかかわらず、初期画面生成(FR-003)は正常に実行される(起動元通知は起動処理のブロッキング条件にしない)。
  - 起動元の値によってグラスの表示内容は変化しない。

### FR-003 初期画面生成

- **概要**: `createStartUpPageContainer()` を**アプリ起動後ちょうど1回**呼び、グラス画面に初期コンテンツを表示する。本案件では feat-002 の時計表示コンテナをそのまま初期画面とする(タイマー本体は後続案件)。
- **入力**: 取得済みブリッジ。
- **出力**: 戻り値が `0`(Success)なら画面表示成功。`0` 以外(1=Invalid/2=Oversize/3=OutOfMemory)ならエラーログを出し、**後始末(cleanup)で登録済みの `onLaunchSource` 購読を解除してから**起動処理を中止し、更新ループを開始しない。
- **受け入れ基準**:
  - シミュレータのグラス画面に初期コンテンツが表示される。
  - 戻り値が `0` 以外の場合、`console.error` に `result` 値と意味が出力され、更新ループ(時計)が始まらず、`onLaunchSource` 購読が解除される(FR-007 の cleanup 経由・購読リークなし)。

### FR-004 ダブルタップによる終了

- **概要**: グラスのダブルタップ(`DOUBLE_CLICK_EVENT`)を受けたら、後始末を実行し `shutDownPageContainer(SHUTDOWN_EXIT_MODE)` でアプリを終了する。`SHUTDOWN_EXIT_MODE = 0`(**即終了・キャンセル不可**)。ダブルタップはタイマーの明確な終了操作であり、確認レイヤを挟まず即座に終了する。即終了のためユーザーがキャンセルする経路は存在せず、終了前に `cleanup()` を実行しても「画面だけ残り更新ループ・購読が解除済み」という復帰不能状態は発生しない。
- **入力**: グラスのダブルタップ操作。
- **出力**: 更新ループ停止・購読解除の後、`shutDownPageContainer(0)` 呼び出し。
- **受け入れ基準**:
  - シミュレータで `POST /api/input {"action":"double_click"}` を送ると、更新ループが停止し(以後スクショの数値が進まない)、cleanup ログが出力される。
  - 終了はキャンセル不可(`exitMode=0`)であり、ダブルタップ後に Running へ復帰する経路は存在しない。

### FR-005 ライフサイクル終了イベントでの後始末

- **概要**: `SYSTEM_EXIT_EVENT`(7)または `ABNORMAL_EXIT_EVENT`(6)を受けたら後始末(更新ループ停止・購読解除)を実行する。
- **入力**: ホストからのライフサイクル終了イベント。
- **出力**: cleanup 実行。
- **受け入れ基準**:
  - 終了イベント受信時に更新ループが停止し、購読が解除される。
  - (注)シミュレータではこれらのイベントは非発火のため、シミュレータ上での自動検証は不可。コードレビューおよび実機(将来)での検証対象とする。

### FR-006 WebView 破棄時の後始末

- **概要**: `window.addEventListener('beforeunload', cleanup)` を登録し、ページ破棄時にも後始末を行う。`cleanup()` 内では登録時と同一の `cleanup` 関数参照で `window.removeEventListener('beforeunload', cleanup)` を実行し、リスナーの多重登録を残さない。
- **入力**: WebView のページアンロード。
- **出力**: cleanup 実行 + `beforeunload` リスナー解除。
- **受け入れ基準**: ページリロード/破棄時に更新ループの `setInterval` が確実に解除される(多重ループが残らない)。cleanup 完了後に `beforeunload` リスナーが残らない。

### FR-007 後始末の冪等性

- **概要**: 後始末(cleanup)は複数回呼ばれても安全(2回目以降は何もしない)であること。FR-004/005/006 が同時/連続で発火しても二重解放やエラーを起こさない。
- **入力**: 複数経路からの cleanup 呼び出し。
- **出力**: 初回のみ資源解放・ログ出力、2回目以降は即 return。
- **受け入れ基準**: ダブルタップ直後に beforeunload が来ても、エラーが出ず cleanup ログは1回のみ。

### FR-008 背面化時の挙動(更新継続)

- **概要**: `FOREGROUND_EXIT_EVENT`(5)で背面化しても更新ループ(`setInterval`)は**停止しない**。`FOREGROUND_ENTER_EVENT`(4)でも特段の再開処理は行わない(止めていないため)。
- **入力**: 前面/背面遷移イベント。
- **出力**: 更新ループは継続。遷移ログの出力は任意(実装する場合 `console.log`)。
- **受け入れ基準**: 背面化イベントを受けても更新ループが止まらず、終了処理(FR-004/005)以外でループが停止しない。
- **備考**: 実機での省電力・BLE 帯域の最適化(背面時にループを止める等)は実機到着後の将来フェーズで再検討する。

## 4. 非機能要求

- **応答性**: ダブルタップから cleanup 完了までの追加遅延は無視できる範囲(ループ停止は同期処理)。
- **対応環境**: シミュレータ(`@evenrealities/evenhub-simulator` 0.7.3)を主検証環境とする。X11(`DISPLAY=:1`)必須。dev server `127.0.0.1:5173`(strictPort)、automation port `9898`。
- **信頼性**: 起動失敗(ブリッジ未取得・画面生成失敗)時もクラッシュせずエラーログを残して停止する。後始末は冪等で、多重ループ・購読リークを残さない。
- **言語/型**: TypeScript `strict: true`。`app/tsconfig.json` に従う。

## 5. 制約条件

- **必須ライブラリ**: `@evenrealities/even_hub_sdk` 0.0.10(`waitForEvenAppBridge`, `onLaunchSource`, `createStartUpPageContainer`, `shutDownPageContainer`, `onEvenHubEvent`, `OsEventTypeList`)。
- **使用禁止**: UI フレームワーク(React/Vue 等)。vanilla TypeScript のみ。
- **ネットワーク**: シミュレータはローカル。外部通信なし。
- **API 制約**: `createStartUpPageContainer()` はアプリ起動後1回のみ有効(2回目以降は無効)。`onLaunchSource` の push はロード完了後最大1回。
- **シミュレータ制約**: launch source の push 有無は未確認。ライフサイクル終了イベントは非発火。
