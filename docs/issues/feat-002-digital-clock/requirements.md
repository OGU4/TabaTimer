# feat-002 要求仕様書: デジタル時計風表示

## 1.1 プロジェクト概要

- **何を作るのか**: Even Hub アプリ `app/` のグラス画面(576×288)に、現在時刻を `HH:MM:SS`(24時間制)で表示し、**毎秒自動更新し続けるデジタル時計**を実装する。
- **なぜ作るのか**: 当面のゴールである「シミュレータ上のインターバルタイマー完成」へ向け、その中核技術である **「同じ位置に表示したまま数値を毎秒切り替え続ける仕組み」(同位置の継続更新)** を最小機能で確立・実証するため。Hello World(feat-001)は静的表示にとどまり、動的更新は未検証だった。
- **誰が使うのか**: 開発者(動作検証目的)。最終的にはトレーニングを行うユーザーだが、本案件は時計表示と更新機構の確立が目的。
- **どこで使うのか**: 開発 PC(Ubuntu 24.04)上の `evenhub-simulator` 0.7.3。実機(Even G2)は将来フェーズ(2026年8月頃見込み)。

## 1.2 用語定義

機能設計書・コード内でも以下の用語を統一して使う。

- **グラス画面**: シミュレータが描画する 576×288 のキャンバス(左上原点、X右+/Y下+)。
- **テキストコンテナ**: SDK の `TextContainerProperty` で定義する文字表示領域。本案件では時刻表示用に1つだけ使う。
- **同位置の継続更新**: テキストコンテナの位置・サイズを変えず、`content`(文字列)だけを差し替えて表示を更新すること。
- **時刻文字列**: 現在時刻を `HH:MM:SS` 形式・24時間制・各フィールド2桁ゼロ埋めで表した8文字の文字列(例: `09:05:30`)。
- **更新ループ**: 一定間隔(1000ミリ秒)ごとに時刻文字列を再生成しテキストコンテナへ反映する処理。
- **automation API**: シミュレータの HTTP API(`--automation-port`)。スクリーンショット・コンソール取得・入力送信に使う。

## 1.3 機能要求一覧

### FR-001 デジタル時計の初期表示

- **機能名**: 起動時の時刻表示
- **概要**: アプリ起動時、グラス画面に現在時刻を `HH:MM:SS`(24時間制)で表示する。
- **入力**: なし(起動時に自動実行)。時刻ソースはシミュレータ実行 PC のローカル時刻(標準 `Date`)。
- **出力**: グラス画面のテキストコンテナに時刻文字列(例 `09:05:30`)が表示される。`createStartUpPageContainer()` の戻り値が `0`(Success)。
- **受け入れ基準**:
  - シミュレータ起動後、`/api/screenshot/glasses` のスクリーンショットに `HH:MM:SS` 形式の8文字(コロン2つ・数字6つ)が描画されている。
  - 表示された時刻が、取得時点のシステム時刻と±2秒以内で一致する。
  - console に `createStartUpPageContainer` 失敗ログ(`result !== 0`)が出ていない。

### FR-002 時刻の毎秒自動更新(同位置の継続更新)

- **機能名**: 毎秒更新
- **概要**: 表示中の時刻を1秒ごとに最新の現在時刻へ自動更新する。テキストコンテナの位置・サイズは変えず `content` のみ差し替える。
- **入力**: なし(更新ループが自動実行)。
- **出力**: テキストコンテナの時刻文字列が秒の進行に追従して更新される。
- **受け入れ基準**:
  - `/api/screenshot/glasses` を3秒以上の間隔をあけて2回取得したとき、表示秒数が経過時間ぶん(±2秒の許容)進んでいる。
  - 更新前後でテキストの表示位置(コンテナの xPosition/yPosition)が変わらない。
  - 各更新で `textContainerUpgrade()` が成功する(戻り値 `true`、または失敗ログが出ない)。

### FR-003 ダブルタップによる終了

- **機能名**: アプリ終了
- **概要**: ユーザーがグラスをダブルタップしたら、更新ループを停止しアプリを終了する。
- **入力**: ダブルタップ操作(automation API では `POST /api/input {"action":"double_click"}`)。
- **出力**: `shutDownPageContainer(1)` が呼ばれ、更新ループ(`setInterval`)が停止し、イベント購読が解除される。
- **受け入れ基準**:
  - `double_click` 送信後、`textContainerUpgrade()` の呼び出しが止まる(console に以降の更新ログが出ない、または `clearInterval` 実行ログが出る)。
  - 終了後にコンソールへ未処理エラー(`[uncaught]` / `[unhandledrejection]`)が出ていない。

## 1.4 非機能要求

- **更新間隔**: 公称1秒(1000ミリ秒)ごとに更新する。実測のずれ許容は±2秒(automation API のポーリング遅延・スクショ取得遅延を考慮)。
- **時刻精度**: 表示時刻はシステム時刻に対し±2秒以内。更新ごとに `Date` を読み直し、`setInterval` の累積ドリフトを表示へ持ち込まない。
- **対応環境**: 開発 PC(Ubuntu 24.04, Node.js v24.11.1)+ `evenhub-simulator` 0.7.3。実機は対象外。
- **滑らかさ**: 毎秒更新時に表示が消える/欠ける時間が連続2回のスクショ観察で発生しないこと(ちらつきの有無はシミュレータで観察し、design.md の「滑らかさ検証」手順で確認する)。
- **信頼性**: `textContainerUpgrade()` が単発で失敗しても更新ループは継続し、次の秒で復帰を試みる(1回の失敗でクラッシュ・停止しない)。

## 1.5 制約条件

- **使用必須**: `@evenrealities/even_hub_sdk` 0.0.10、Vite 8.x、TypeScript 6.x(feat-001 の `app/` 構成を継続)。新規ライブラリは導入しない。
- **UIフレームワーク禁止**: vanilla TypeScript のまま実装する(CLAUDE.md 技術スタック方針)。
- **コンテナ制約**: テキストコンテナは1つ。ちょうど1つに `isEventCapture: 1` を設定する(SDK 制約)。
- **content 長**: 時刻文字列は8バイト。シミュレータのテキスト 999 バイト上限・SDK の `containerName` 16文字上限を超えない。
- **画面更新手段**: 初回は `createStartUpPageContainer()`(1回のみ)、以降は `textContainerUpgrade()` を使う(`createStartUpPageContainer` の再呼び出しは無効)。
- **時刻ソース**: ネットワーク時刻同期は行わない。実行 PC のローカル時刻(標準 `Date`)のみ使用。
- **ネットワーク**: オフライン動作(外部 API 呼び出しなし)。

## 1.6 優先順位(MoSCoW)

- **Must**: FR-001(初期表示)、FR-002(毎秒更新)。この2つが本案件の MVP かつ目的そのもの。
- **Should**: FR-003(ダブルタップ終了)。公式テンプレート慣習に沿った後始末。実装容易かつ安全性に寄与するため本案件に含める。
- **Could**: なし。
- **Won't(本案件では扱わない)**:
  - 12時間制・AM/PM 表示、日付・曜日表示。
  - 時刻のタイムゾーン選択・手動設定。
  - フォントサイズ/レイアウトのカスタマイズ。
  - インターバルタイマー機能(運動/休憩フェーズ、ラウンド)。これは後続案件。
  - 実機での動作確認(実機到着後の将来フェーズ)。
